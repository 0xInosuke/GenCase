# AI Search Module

This project includes an independent AI-assisted case search module for the case list page.

The module does not change the existing database schema or the normal CRUD/search behavior. It extends the unified case-list search field so a user can type natural language and GenCase converts that request into a structured case-search plan.

## Configuration

Add the following values to `.env`:

```text
AI_API_URL=https://your-llm-endpoint.example/v1/chat/completions
AI_API_KEY=your_api_key
AI_MODEL=your_model_name
AI_TIMEOUT_MS=20000
AI_RATE_LIMIT_WINDOW_MS=60000
AI_RATE_LIMIT_MAX_REQUESTS=20
AI_SEMANTIC_FILTER_ENABLED=false
AI_SEMANTIC_CANDIDATE_LIMIT=120
```

These settings are independent from the main database and app configuration.

`AI_API_URL` must be the final provider endpoint. GenCase does not rewrite or append provider paths automatically.

`AI_SEMANTIC_FILTER_ENABLED` is the switch for the new hybrid semantic filter:

- `false`: keep the existing plan-based AI search only
- `true`: first run candidate recall locally, then let the LLM return matching `case_id` values from candidate summaries

`AI_SEMANTIC_CANDIDATE_LIMIT` controls how many candidate case summaries are sent to the LLM in semantic mode.

## Flow

1. The user types into the unified case-list search field.
2. The frontend automatically decides whether to use:
   - normal text search
   - JSON condition search
   - AI-assisted natural-language search
3. While the AI request is running, the UI shows a busy indicator so the user can tell the search is still in progress.
4. The user can enter a natural-language request such as:
   - `give me all cases that risk score greater than 5`
   - `list all cases assigned to bob or bob works on`
   - `all cases alice replied`
5. If semantic filtering is disabled, the backend sends the prompt plus a compact GenCase search design guide and observed visible-case context to the configured LLM.
6. If semantic filtering is enabled, the backend first builds a local snapshot cache of visible cases, selects a limited candidate set, and sends only candidate summaries to the LLM.
7. Visible-case context may include:
   - observed workflow names
   - observed stage codes
   - observed `case_data` JSON paths
   - observed comment authors and user names
8. In plan mode, the LLM must return strict JSON containing:
   - a short `explanation`
   - a structured `plan`
9. In semantic mode, the LLM must return a list of matching candidate `case_id` values only.
10. GenCase validates the returned plan or case IDs before executing anything.
11. In semantic mode, GenCase never trusts provider-side memory; it keeps a local snapshot cache keyed by user visibility and rebuilds when visible data changes.
12. If the first plan-mode result returns zero results, GenCase retries once with a broader context prompt based on the visible-case context.
13. Results are returned to the frontend using the same pagination and sorting style as the normal case list.

## Supported Search Plan

Allowed fields:

- `case_title`
- `wf_name`
- `stage_code`
- `last_edited_by`
- `comment_authors`
- `comment_user_names`
- `comment_contents`
- `case_data.<json_path>`

Allowed operators:

- `eq`
- `contains`
- `gt`
- `gte`
- `lt`
- `lte`
- `in`

The AI module does not execute raw SQL generated by the model.

## Semantic Filter Mode

When `AI_SEMANTIC_FILTER_ENABLED=true`, GenCase uses a hybrid flow:

1. Load visible cases for the current user.
2. Build cached lightweight summaries in memory.
3. Rank and trim candidates locally.
4. Send only candidate summaries to the LLM.
5. Accept only matching `case_id` values that already exist in the candidate set.
6. Rehydrate those IDs from local data and return them to the UI.

This avoids sending the entire visible dataset on every request and avoids relying on long-lived model memory.

## API

Authenticated session route:

```text
POST /api/ai/case-search
```

Request body supports either:

```json
{
  "prompt": "give me all cases that risk score greater than 5",
  "page": 1,
  "page_size": 20,
  "sort_by": "id",
  "sort_dir": "asc"
}
```

or an already generated structured plan:

```json
{
  "plan": {
    "match": "and",
    "conditions": [
      { "field": "case_data.metadata.score", "operator": "gt", "value": 5 }
    ]
  },
  "page": 1,
  "page_size": 20,
  "sort_by": "id",
  "sort_dir": "asc"
}
```

## Safety Rules

- AI search only returns cases already visible to the current logged-in user.
- The AI route is mounted under `/api`, so requests without a valid session cookie are rejected with `401`.
- There is no external API-key variant of the AI route. `/external-api/...` does not expose AI search.
- The model may suggest filters, but GenCase executes them locally after validation.
- Unsupported fields and operators are rejected before execution.
- AI prompt length is capped to reduce accidental oversized requests.
- AI requests are rate-limited per logged-in user to reduce abuse and uncontrolled model spend.
- Existing `/api/cases` list search remains unchanged.

## Abuse Risk Notes

Current risk profile:

- Unauthenticated internet callers cannot use the AI route because `/api/ai/case-search` requires a login session.
- Authenticated users can still call the route directly with browser devtools, `curl`, or custom scripts. That is expected because the frontend itself uses the same backend API.
- The main remaining abuse risk is cost or volume from a valid logged-in user repeatedly sending prompts.

Current protections:

- Session authentication is required.
- `SameSite=Lax` and `HttpOnly` are set on the session cookie.
- Requests are rate-limited per user.
- The backend validates the returned plan before executing it.
- The backend never executes model-generated SQL.

## Suggested Manual API Checks

1. Unauthenticated request should fail:

```http
POST /api/ai/case-search
```

Expected:

```json
{ "error": "Authentication required." }
```

2. Authenticated request with a natural-language prompt should succeed.
   If semantic mode is enabled, the response `ai.mode` should be `semantic` when the LLM returns matched `case_id` values.

3. Authenticated request with an oversized prompt should return `400`.

4. Repeated authenticated requests above the configured limit should return `429`.
